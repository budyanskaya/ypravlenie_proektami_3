3. Оценка и распределение ресурсов

# 1. МЕТОД ОЦЕНКИ ПОТРЕБНОСТИ В РЕСУРСАХ
Для проекта с ограниченными ресурсами и бюджетом рекомендуется использовать комбинированный метод «Снизу вверх» (Bottom-Up) с экспертной оценкой.
 
Алгоритм:
1. Декомпозиция работ (WBS): Разбить каждую из 5 задач на мелкие операции.
2. Определение компетенций: Для каждой операции четко указать требуемую роль (Аналитик, Dev, QA, Design).
3. Оценка трудозатрат: Экспертным путем оценить, сколько человеко-часов нужно на операцию.
4. Учет доступности:
   - Сотрудник А (50% загрузка) = 4 часа/день на проект.
   - Сотрудник Б (75% загрузка) = 6 часов/день на проект.
5. Расчет длительности:
Например: 32 часа работы / 4 часа в день = 8 календарных дней.
 
Почему этот метод? Он дает наибольшую точность при ограниченных ресурсах, так как учитывает реальную пропускную способность команды, а не идеальные условия.
 
# 2. ИНСТРУМЕНТ ДЛЯ ОПТИМАЛЬНОГО РАСПРЕДЕЛЕНИЯ
Инструмент: Вид «Использование ресурсов» в ProjectLibre с функцией построения гистограммы загрузки + анализ критического пути.
 
Как оптимизировать распределение в ProjectLibre.
- Учет компетенций через назначение ресурсов:
1.	В таблице ресурсов задайте каждому сотруднику роль и максимальную доступность (50% или 75%).
2.	При назначении задач в диаграмме Ганта выбирайте сотрудника, чья компетенция соответствует задаче.
- Приоритизация через Критический путь:
Включите отображение критического пути в диаграмме Ганта (красные задачи).
Задачи с резервом времени (Float) можно перемещать в пределах этого резерва, чтобы разгрузить сотрудников в пиковые периоды.
- Выравнивание нагрузки через «Использование ресурсов»:
1.	Перейдите в вид «Использование ресурсов» (вкладка «Просмотр» → «Использование ресурсов»).
2.	Разверните список под каждым сотрудником — увидите почасовую/по дням загрузку.
Нажмите кнопку «Гистограмма»  для визуального отображения пиков нагрузки.
 
# 3. КОНТРОЛЬ ПЕРЕГРУЗКИ И МИНИМИЗАЦИЯ ПРОСТОЕВ
Контроль перегрузки (Overload):
1. Визуальный мониторинг: Еженедельный просмотр Resource Histogram. Если столбец загрузки превышает 100% от доступных единиц (Max Units) — это перегрузка.
2. Правило «Красной зоны»:
   - Если загрузка > 85% от доступного времени (например, сотрудник работает 6.5 часов из 8 доступных), риск срыва сроков резко возрастает.
   - Действие: Перераспределить часть задач на другого сотрудника или увеличить длительность задачи.
 
Минимизация простоев:
1. Предварительное планирование:
   - Планировать задачи на 2–3 недели вперед. Если видно, что Дизайнер свободен на следующей неделе, а Разработчик перегружен, можно ли начать подготовку макетов для следующего этапа раньше?
2. Гибкие роли:
   - Использовать универсальных сотрудников (как Иванов в примере) для заполнения «дыр» в графике узких специалистов.
3. Параллелизация задач:
   - Если есть простой у Тестировщика, а Разработчик еще пишет код, можно начать писать тест-кейсы по готовым требованиям, не дожидаясь конца разработки.
